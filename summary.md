## 1. Постановка задачи

Проект сравнивает классические ML-модели для молекулярных дескрипторов:

- регрессия: `IC50, mM`, `CC50, mM`, `SI`;
- классификация: `IC50 > median`, `CC50 > median`, `SI > median`, `SI > 8`.

После диагностики data leakage принят строгий **descriptor-only** подход:
из матрицы признаков `X` всегда удаляются все endpoint/target-like колонки:

- `IC50, mM`, `CC50, mM`, `SI`;
- `IC50_log`, `CC50_log`, `SI_log`;
- любые колонки с `ic50` или `cc50` в имени;
- точные SI-производные: `SI`, `SI_log`, `log_SI`, `SI_bin`,
  `SI_class`, `SI_scaled`, `target_SI`.

Такая схема делает регрессию и классификацию сопоставимыми: модели используют
только молекулярные дескрипторы, а не экспериментальные endpoints или их
производные.

## 2. Исправление data leakage и корректировка результатов

### Где была утечка

В notebook workflow после EDA создавалась колонка:

```python
df["IC50_log"] = np.log10(df["IC50, mM"] + 1e-8)
```

Затем `get_feature_columns()` удалял только часть исходных targets, но не
удалял `IC50_log`. В результате для задачи `IC50 > median` модель получала
монотонное преобразование целевой переменной прямо в признаках.

### Почему возникал ROC-AUC около 1.0

Классификационный target:

```python
(df["IC50, mM"] > median(IC50)).astype(int)
```

почти полностью восстанавливается из признака:

```python
IC50_log = log10(IC50 + eps)
```

Деревья находили порог по `IC50_log`, а логистическая регрессия использовала
почти линейно отделимый признак. Поэтому ROC-AUC 0.99-1.0 был не качеством
модели, а прямым следствием утечки.

### Почему это нереалистично

В реальной задаче drug discovery на этапе предсказания значения `IC50`,
`CC50` и `SI` неизвестны: именно их и требуется прогнозировать. Если эти
величины или их логарифмы попадают в `X`, модель решает не задачу прогноза,
а задачу восстановления уже известного ответа.

### Как leakage устранен

Исправление внесено системно:

- `ml_common.py` содержит единый список target-like колонок;
- `build_feature_matrix()` удаляет все target-like признаки;
- `validate_no_target_like_columns()` выбрасывает `ValueError`, если leakage
  обнаружен перед обучением;
- `classification_utils.py` и `regression_utils.py` вызывают runtime-проверку
  до split/fit;
- notebook использует ту же логику фильтрации;
- stale notebook outputs очищены, чтобы старые leakage-метрики не выглядели
  как актуальные результаты.

### Какие метрики стали после исправления

В этой среде post-fix метрики не пересчитаны, потому что отсутствует файл:

```text
data/Данные_для_курсовой_Классическое_МО.xlsx
```

Команды для честного пересчета:

```bash
python3 classification_ic50.py
python3 classification_cc50.py
python3 classification_si.py
```

После появления датасета эти команды обновят CSV в `results/`. Метрики с
ROC-AUC около 0.99-1.0 из старого notebook не должны использоваться в выводах.

## 4. Ожидаемая интерпретация после честного пересчета

Для классификации сравниваются:

- Logistic Regression;
- Random Forest Classifier;
- Gradient Boosting Classifier.

Основная метрика выбора: **ROC-AUC**. Дополнительно важны accuracy, precision,
recall, F1 и PR-AUC, особенно для `SI > 8`, где классы несбалансированы.

Ожидаемо, что Random Forest / boosting будут сильнее Logistic Regression,
потому что:

- химические дескрипторы взаимодействуют нелинейно;
- деревья лучше моделируют пороги и локальные области пространства признаков;
- ансамбли устойчивее к выбросам и тяжелым хвостам;
- логистическая регрессия остается полезным baseline, но ограничена линейной
  границей решения.

## 5. Ограничения исследования

1. **Нет внешней валидации.** Hold-out split показывает внутреннюю
   обобщающую способность, но не гарантирует переносимость на новые серии
   соединений.
2. **Малый размер данных.** Около 1000 объектов недостаточно для надежной
   оценки редких высокоактивных/высокотоксичных областей.
3. **SI тяжелохвостый.** Метрики для SI-задач чувствительны к выбросам и
   выбранному порогу.
4. **Пороговые задачи упрощают биологию.** `> median` удобно для сравнения
   моделей, но не всегда соответствует практическому decision threshold.

## 6. Рекомендации

### Какие модели использовать дальше

- Для базового production-like baseline: **Random Forest**.
- Для проверки потенциального прироста: **Gradient Boosting** и современные
  бустинги (`LightGBM`, `XGBoost`, `CatBoost`).
- Logistic Regression оставить как интерпретируемый baseline и sanity-check.

### Что улучшать

- repeated stratified CV и доверительные интервалы метрик;
- PR-AUC и calibration curves для несбалансированных задач;
- permutation importance / SHAP для отбора признаков;
- устойчивую обработку выбросов без удаления биологически важных экстремумов;
- внешнюю тестовую выборку или scaffold split, если доступны структуры.

### Feature engineering

- агрегаты по группам дескрипторов;
- PCA/UMAP только внутри cross-validation pipeline;
- отбор коррелированных признаков;
- нелинейные взаимодействия дескрипторов без использования endpoints.
