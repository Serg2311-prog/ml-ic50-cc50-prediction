# Аналитический отчет по проекту

## 1. Постановка задачи

Цель проекта - построить и сравнить классические модели машинного обучения для
прогноза биологических endpoints по молекулярным дескрипторам. Рассматриваются:

- регрессия `IC50, mM`;
- регрессия `CC50, mM`;
- регрессия `SI`;
- классификация `IC50 > median`;
- классификация `CC50 > median`;
- классификация `SI > median`;
- классификация `SI > 8`.

Работа ориентирована не только на получение метрик, но и на корректную
исследовательскую постановку: исключение data leakage, интерпретацию EDA,
сравнение моделей, анализ ограничений и формулировку практических рекомендаций.

## 2. Данные и воспроизводимость

Датасет должен располагаться по стабильному пути:

```text
data/Данные_для_курсовой_Классическое_МО.xlsx
```

Все скрипты используют общий загрузчик из `ml_common.py`, что делает запуск после
`git clone` воспроизводимым. При чтении удаляется техническая колонка
`Unnamed: 0`, если она присутствует.

После удаления служебного индекса данные содержат 1001 объект и 213 колонок:
3 endpoint-колонки (`IC50, mM`, `CC50, mM`, `SI`) и набор числовых молекулярных
дескрипторов.

## 3. EDA

### 3.1 Качество данных

EDA показывает, что датасет в целом пригоден для классического ML:

- число объектов достаточно для baseline-сравнения моделей;
- пропусков немного, поэтому медианная импутация является разумным базовым
  вариантом;
- признаки числовые и совместимы с линейными моделями, деревьями и бустингом;
- техническая индексная колонка удаляется до моделирования.

Медианная импутация предпочтительна для этого датасета, потому что многие
дескрипторы и endpoints имеют асимметричные распределения. Среднее значение в
такой ситуации сильнее смещается выбросами.

### 3.2 Распределения endpoints

Целевые переменные имеют выраженную асимметрию:

- `IC50` - правосторонняя асимметрия, много умеренных значений и небольшой
  хвост крупных значений;
- `CC50` - также асимметричен, но менее экстремален, чем `SI`;
- `SI` - самый тяжелохвостый target, так как является отношением
  `CC50 / IC50`.

Из-за тяжелых хвостов простые линейные модели не всегда адекватно описывают
зависимость между дескрипторами и endpoints. Это заранее объясняет, почему
ансамбли деревьев показывают более устойчивое качество.

### 3.3 Выбросы

Выбросы обнаруживаются в endpoints и части дескрипторов. Они не удалялись
механически, потому что в drug discovery экстремальные значения могут быть
биологически значимыми: высокоселективные или высокотоксичные соединения часто
как раз находятся в хвостах распределений.

Корректная стратегия для будущей работы:

- анализировать выбросы отдельно;
- проверять robust-scaling или winsorization только внутри CV;
- не удалять экстремальные значения без химико-биологического обоснования.

### 3.4 Корреляции и мультиколлинеарность

Для молекулярных дескрипторов ожидаема мультиколлинеарность: многие признаки
описывают близкие физико-химические свойства молекулы. Это важно для выбора
моделей:

- Logistic/Linear Regression чувствительнее к коррелированным признакам и
  масштабам;
- Random Forest и Gradient Boosting устойчивее к коррелированным группам, но
  importance может распределяться между похожими признаками;
- для интерпретации в будущем полезны permutation importance, SHAP и отбор
  коррелированных признаков.

### 3.5 Связь признаков с target

EDA показывает, что сигнал в дескрипторах есть, но он не является идеально
линейным. Для `CC50` связь с дескрипторами наиболее стабильная; для `SI` задача
сложнее, потому что `SI` является производным отношением и имеет тяжелый хвост.

Отсюда следует важный вывод: ROC-AUC около 0.7-0.8 для classification задач
после удаления leakage выглядит реалистично, а ROC-AUC около 1.0 - нет.

### 3.6 Проверка логики SI и leakage-опасные признаки

`SI` связан с `IC50` и `CC50` как производный endpoint. Поэтому:

- при прогнозировании `SI` нельзя использовать `IC50` и `CC50`;
- при прогнозировании `IC50`/`CC50` нельзя использовать `SI`;
- при descriptor-only постановке безопаснее удалить все endpoints из `X` для
  любой задачи.

В финальной версии выбран именно descriptor-only подход: из признаков всегда
удаляются `IC50`, `CC50`, `SI` и их производные target-like колонки.

## 4. Исправление data leakage и корректировка результатов

### 4.1 Где была утечка

В notebook workflow после EDA создавалась колонка:

```python
df["IC50_log"] = np.log10(df["IC50, mM"] + 1e-8)
```

Затем при формировании признаков удалялись не все производные endpoints.
`IC50_log` оставался в `X`. Для задачи `IC50 > median` это почти прямое
раскрытие ответа, потому что логарифм является монотонным преобразованием
`IC50`.

### 4.2 Почему возникал ROC-AUC около 1.0

Target `IC50 > median` почти полностью восстанавливается правилом:

```text
IC50_log > log10(median(IC50) + eps)
```

Деревья легко находят такой порог, а логистическая регрессия получает почти
линейно отделимый признак. Поэтому ROC-AUC `0.99-1.00` был не качеством модели,
а следствием data leakage.

### 4.3 Почему это нереалистично

В production-like сценарии значения `IC50`, `CC50` и `SI` неизвестны на момент
предсказания. Если endpoint или его логарифм попадает в признаки, модель не
прогнозирует биологическое свойство по дескрипторам, а использует уже известный
ответ. Такое качество не переносится на новые молекулы.

### 4.4 Как leakage устранен

Исправление сделано системно:

- в `ml_common.py` задан общий список endpoint/target-like колонок;
- `get_target_like_columns()` ловит явные targets и производные по имени;
- `build_feature_matrix()` удаляет все target-like колонки;
- `validate_no_target_like_columns()` выбрасывает `ValueError`, если leakage
  обнаружен перед обучением;
- classification и regression pipelines используют эту проверку;
- notebook outputs очищены от старых leakage-результатов.

Для `SI` используется аккуратная логика: не удаляются все признаки с подстрокой
`si`, потому что это могло бы повредить обычные дескрипторы; удаляются только
точные target-like варианты (`SI`, `SI_log`, `log_SI`, `SI_bin`, `SI_class`,
`SI_scaled`, `target_SI`).

### 4.5 Результаты после исправления

После удаления leakage classification ROC-AUC стали реалистичными:

| Задача | Logistic Regression | Random Forest | Gradient Boosting | Лучшая модель |
|---|---:|---:|---:|---|
| IC50 > median | 0.75 | **0.78** | 0.77 | Random Forest |
| CC50 > median | **0.83** | **0.83** | 0.81 | Random Forest / Logistic Regression |
| SI > median | 0.65 | **0.69** | 0.67 | Random Forest |
| SI > 8 | 0.67 | **0.75** | 0.74 | Random Forest |

Эти значения научно корректнее, потому что модели используют только
молекулярные дескрипторы и не имеют доступа к экспериментальным endpoints.

## 5. Regression analysis

### 5.1 IC50 regression

| Модель | RMSE | MAE | R2 |
|---|---:|---:|---:|
| Gradient Boosting | **415.49** | 208.81 | **0.4825** |
| Random Forest | 418.91 | **206.03** | 0.4739 |
| Linear Regression | 441.22 | 245.14 | 0.4164 |

Лучшая модель по RMSE - Gradient Boosting. Она лучше улавливает нелинейные
зависимости между дескрипторами и активностью. Random Forest близок по качеству
и имеет меньший MAE, что делает его надежным baseline. Linear Regression
уступает из-за нелинейности и влияния тяжелых хвостов.

### 5.2 CC50 regression

| Модель | RMSE | MAE | R2 |
|---|---:|---:|---:|
| Random Forest | **415.75** | **235.86** | **0.6666** |
| Gradient Boosting | 427.01 | 246.49 | 0.6483 |
| Linear Regression | 446.26 | 325.56 | 0.6159 |

`CC50` прогнозируется лучше, чем `IC50` и `SI`: R2 выше, а ансамбли деревьев
дают устойчивое качество. Это говорит о более стабильной связи токсичности с
дескрипторами в данном наборе.

### 5.3 SI regression

| Модель | RMSE | MAE | R2 |
|---|---:|---:|---:|
| Gradient Boosting | **1353.47** | 198.30 | **0.0880** |
| Random Forest | 1358.54 | **191.30** | 0.0812 |
| Linear Regression | 1379.37 | 222.30 | 0.0528 |

Регрессия `SI` остается слабой у всех моделей. Это ожидаемо: `SI` является
отношением двух endpoints и имеет экстремальный хвост. Для этой задачи лучше
рассматривать log-transform, robust losses, ранжирование или классификационные
пороговые постановки.

## 6. Classification analysis

### 6.1 IC50 > median

| Модель | ROC-AUC |
|---|---:|
| Random Forest | **0.78** |
| Gradient Boosting | 0.77 |
| Logistic Regression | 0.75 |

Random Forest является лучшим вариантом: он устойчивее к нелинейным порогам в
пространстве дескрипторов. Разрыв с Gradient Boosting небольшой, поэтому обе
модели подходят для дальнейшего сравнения.

### 6.2 CC50 > median

| Модель | ROC-AUC |
|---|---:|
| Random Forest | **0.83** |
| Logistic Regression | **0.83** |
| Gradient Boosting | 0.81 |

`CC50` - наиболее стабильная classification задача. Logistic Regression почти
догоняет Random Forest по ROC-AUC, что указывает на наличие достаточно сильного
линейного компонента. Тем не менее Random Forest предпочтителен как более
гибкая production-like модель.

### 6.3 SI > median

| Модель | ROC-AUC |
|---|---:|
| Random Forest | **0.69** |
| Gradient Boosting | 0.67 |
| Logistic Regression | 0.65 |

Качество ниже, чем для `IC50` и `CC50`, потому что `SI` - производный и
тяжелохвостый endpoint. Random Forest остается лучшим, но метрики показывают,
что задача требует дополнительного feature engineering.

### 6.4 SI > 8

| Модель | ROC-AUC |
|---|---:|
| Random Forest | **0.75** |
| Gradient Boosting | 0.74 |
| Logistic Regression | 0.67 |

Для `SI > 8` PR-AUC особенно важен из-за дисбаланса классов, поэтому при
полном rerun его нужно анализировать вместе с ROC-AUC. По доступному ROC-AUC
Random Forest лучше ранжирует соединения по вероятности высокого SI, а
Gradient Boosting остается близкой альтернативой.

## 7. Сравнение моделей

### Почему Random Forest / Gradient Boosting сильнее Logistic Regression

Ансамбли деревьев лучше соответствуют природе задачи:

- молекулярные дескрипторы взаимодействуют нелинейно;
- часть зависимостей пороговая;
- распределения endpoints асимметричны;
- выбросы могут быть информативными;
- коррелированные признаки мешают линейным коэффициентам, но меньше вредят
  ансамблям.

Logistic Regression полезна как baseline и sanity-check. Если она показывает
качество, близкое к деревьям, это говорит о наличии линейного сигнала. Если она
внезапно показывает ROC-AUC около 1.0, это повод искать leakage.

## 8. Ограничения исследования

1. **Внутренняя валидация.** Использован hold-out и CV, но нет внешнего набора.
2. **Размер данных.** Около 1000 объектов ограничивают устойчивость оценки.
3. **Тяжелые хвосты.** Особенно критично для `SI`.
4. **Пороговые targets.** `median` удобна для сравнения, но не всегда
   соответствует практическому биологическому threshold.
5. **Нет scaffold split.** При наличии структур молекул это важный следующий
   шаг для проверки обобщения на новые химические классы.

## 9. Рекомендации

### Модели для дальнейшего использования

- Основной baseline: Random Forest.
- Дополнительный кандидат: Gradient Boosting.
- Интерпретируемый baseline: Logistic Regression.
- Для следующего этапа: LightGBM, XGBoost, CatBoost.

### Feature engineering

- группировка дескрипторов по физико-химическому смыслу;
- удаление или агрегация сильно коррелированных признаков;
- PCA только внутри pipeline/CV;
- SHAP/permutation importance для отбора признаков;
- log-transform endpoints для regression;
- calibration curves для classification.

### Валидация

- repeated stratified CV;
- confidence intervals для ROC-AUC/PR-AUC;
- scaffold split;
- внешний hold-out набор;
- анализ ошибок по химическим подгруппам.

## 10. Практические выводы

После исправления leakage проект показывает реалистичную картину:

- `CC50` прогнозируется наиболее стабильно;
- `IC50` имеет умеренно предсказуемый сигнал;
- `SI` остается наиболее сложной задачей;
- Random Forest - лучший общий выбор для production-like baseline;
- старые ROC-AUC около 1.0 были ошибочными и полностью исключены из финальных
  результатов.

Главная ценность работы - не только сравнение моделей, но и демонстрация
корректной ML-методологии: EDA, диагностика leakage, reproducible pipeline,
сравнение моделей и осторожная интерпретация результатов.
