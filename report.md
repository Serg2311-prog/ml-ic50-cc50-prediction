# Аналитический отчет по проекту IC50 / CC50 / SI

## 1. Цель исследования

Проект сравнивает классические модели машинного обучения для прогнозирования
биологических endpoints по молекулярным дескрипторам:

- регрессия: `IC50, mM`, `CC50, mM`, `SI`;
- классификация: `IC50 > median`, `CC50 > median`, `SI > median`, `SI > 8`.

После диагностики проекта основной методологический акцент перенесен на
корректную постановку задачи без data leakage. Все модели должны использовать
только descriptor-only признаки: экспериментальные endpoints и их производные
не допускаются в матрицу признаков `X`.

## 2. Исправление data leakage и корректировка результатов

### 2.1 Где была утечка

В notebook workflow после EDA создавалась колонка:

```python
df["IC50_log"] = np.log10(df["IC50, mM"] + 1e-8)
```

Далее при формировании признаков удалялись не все target-like колонки. В
частности, `IC50_log` оставался в `X`. Для задачи `IC50 > median` это является
прямой утечкой: бинарный target почти полностью восстанавливается порогом по
монотонному преобразованию `IC50`.

### 2.2 Почему ROC-AUC около 1.0 был ошибочным

Если модель получает `IC50_log`, то задача `IC50 > median` становится почти
тривиальной:

```text
IC50 > median(IC50)
```

эквивалентно пороговому правилу по:

```text
log10(IC50 + eps)
```

Поэтому Random Forest, Extra Trees, Gradient Boosting и даже Logistic
Regression могли показывать ROC-AUC около `0.99-1.0`. Это не отражало
предсказательную способность модели на новых молекулах, а означало, что в
признаки попала информация о целевой переменной.

### 2.3 Почему это нереалистично

В прикладной задаче drug discovery значения `IC50`, `CC50` и `SI` неизвестны
до эксперимента. Если использовать эти величины или их производные как
признаки, модель не предсказывает endpoint, а восстанавливает уже известный
ответ. Такие метрики нельзя использовать для выбора модели или научных
выводов.

### 2.4 Как leakage устранен

Исправление внесено системно:

- датасет загружается из стабильного пути
  `data/Данные_для_курсовой_Классическое_МО.xlsx`;
- `ml_common.py` содержит единый список target-like колонок;
- `get_target_like_columns()` дополнительно ловит производные по именам:
  - любые колонки с `ic50` в названии;
  - любые колонки с `cc50` в названии;
  - ограниченный набор SI-производных (`SI`, `SI_log`, `log_SI`,
    `SI_bin`, `SI_class`, `SI_scaled`, `target_SI`);
- `build_feature_matrix()` удаляет все найденные target-like колонки;
- `validate_no_target_like_columns()` выбрасывает `ValueError`, если leakage
  обнаружен перед обучением;
- classification и regression pipelines используют runtime-проверку перед
  fit;
- notebook outputs очищены, чтобы старые leakage-таблицы не противоречили
  исправленному pipeline.

### 2.5 Статус метрик после исправления

В текущей среде исходный Excel-файл отсутствует, поэтому post-fix метрики не
могут быть честно пересчитаны. Файл:

```text
data/Данные_для_курсовой_Классическое_МО.xlsx
```

должен быть добавлен локально перед запуском экспериментов.

До повторного запуска все старые численные результаты помечены как требующие
регенерации. Метрики ROC-AUC около `0.99-1.0` считаются leakage-артефактами и
исключены из актуальных выводов.

## 3. Как получить корректные classification results

После размещения датасета нужно выполнить:

```bash
python3 classification_ic50.py
python3 classification_cc50.py
python3 classification_si.py
```

Classification pipeline сохраняет:

- accuracy;
- precision;
- recall;
- F1;
- ROC-AUC;
- PR-AUC;
- лучшие гиперпараметры.

Если после исправления leakage Logistic Regression или деревья снова покажут
ROC-AUC около `0.99-1.0`, это должно рассматриваться как сигнал к повторной
диагностике leakage или дубликатов/near-duplicates в данных.

## 4. Ожидаемая интерпретация моделей

### Logistic Regression

Логистическая регрессия полезна как baseline:

- быстро обучается;
- хорошо интерпретируется;
- показывает, насколько задача решается линейной границей.

Ограничение: химические дескрипторы часто взаимодействуют нелинейно, поэтому
линейная модель обычно уступает ансамблям деревьев по ROC-AUC.

### Random Forest

Random Forest является сильным baseline для дальнейшей работы:

- моделирует нелинейные пороги;
- устойчив к выбросам;
- меньше зависит от масштабирования;
- хорошо работает на табличных дескрипторах среднего размера.

### Gradient Boosting

Boosting полезен как кандидат на лучшую модель при тщательном подборе
гиперпараметров:

- строит последовательные корректировки ошибок;
- часто лучше улавливает слабые нелинейные сигналы;
- может переобучаться сильнее Random Forest, поэтому требует аккуратной
  валидации.

## 5. Ограничения исследования

1. **Отсутствие внешней валидации.** Hold-out split не гарантирует
   переносимость на новые химические серии.
2. **Размер выборки.** Около тысячи объектов ограничивает надежность выводов,
   особенно для редких высокоактивных соединений.
3. **Тяжелые хвосты endpoints.** `SI` особенно чувствителен к выбросам и
   выбранному порогу.
4. **Пороговые classification задачи.** `> median` удобно для сравнения
   моделей, но не всегда соответствует практическому biological threshold.
5. **Риск новых target-like колонок.** Если в датасет добавятся новые
   производные endpoints с нестандартными именами, их нужно включить в
   detector.

## 6. Рекомендации для дальнейшей работы

- Использовать Random Forest как основной robust baseline.
- Сравнить его с Gradient Boosting и современными boosting-библиотеками
  (`LightGBM`, `XGBoost`, `CatBoost`).
- Для classification задач дополнительно анализировать PR-AUC и calibration
  curves.
- Использовать repeated stratified CV и доверительные интервалы.
- Добавить scaffold split или внешний hold-out, если доступны структуры
  молекул.
- Проводить feature selection через permutation importance или SHAP.
- Делать PCA/UMAP и другие трансформации только внутри CV pipeline, чтобы не
  добавить новую утечку.
- Для `SI` отдельно исследовать хвосты распределения и несколько
  practically meaningful thresholds (`SI > 8`, `SI > 10`, etc.).

## 7. Финальный вывод

Проект методологически усилен: основная проблема data leakage диагностирована,
объяснена и закрыта системно. До повторного запуска на исходном датасете
финальные численные метрики не должны декларироваться. После размещения
датасета в `data/` текущий pipeline позволяет получить согласованные и научно
корректные classification/regression results без использования endpoints в
признаках.
